# Пайплайн: Opus 4.8 (оркестратор) + Cursor CLI / Composer 2.5 (исполнитель)

План создания, чеклист, начальные тесты и постоянные метрики.
Документ собран по итогам всех обсуждённых решений — читать сверху вниз, строить по фазам.

---

## 0. Базовые принципы (зафиксированы, не пересматриваются на ходу)

1. **Composer 2.5 не имеет внешнего API.** Достижим только через бинарь `cursor-agent` (headless `-p`). Поэтому Opus не вызывает Composer как API-субагента — Opus через bash дёргает Cursor CLI и забирает JSON.
2. **Отчёт Composer — это недоверенные данные.** Opus проверяет ground truth (git diff, вывод тестов, тайпчекер/линтер), а не нарратив исполнителя.
3. **Источник правды — диск, не окно контекста модели.** git + тесты = жёсткое состояние; `STATE.md` + ledger = мягкое состояние, автор которого только Opus.
4. **Критичные договорённости — инварианты пайплайна, а не текстовые правила.** Тест каждого пункта: «если модель это проигнорирует, я замечу автоматически или только вручную залезши в код?» Если только вручную — переносим из текста в структуру.
5. **Экономия не предполагается, а измеряется.** Множитель токенов у агентных схем реален (агенты ~4x, мультиагент ~15x против чата), и кодинг — плохой кандидат на наивную мультиагентность. Выигрыш существует, только если грязная работа доминирует, а контекст оркестратора остаётся тонким.

**Асимметрия ролей:**
- Opus (доверенный слой) — правила про *дисциплину делегирования*: не кодить самому, формулировать машинно-проверяемое «готово», не раздувать собственный контекст.
- Composer (недоверенный слой) — правила про *границы*: что можно трогать, что вернуть, как честно сообщить о провале.

## 1. Целевая архитектура (одной картинкой)

```
[Ты] → Opus 4.8 в Claude Code (оркестратор, на подписке)
            │  читает STATE.md + ledger + сверку (git/тесты)
            │  пишет ТЗ с машинно-проверяемыми критериями приёмки
            ▼
      [твой клей: bash-обёртка]
            │  создаёт git worktree, вызывает:
            │  cursor-agent -p --force --model composer --output-format json "<ТЗ>"
            ▼
   Composer 2.5 в Cursor CLI (исполнитель, дешёвые токены/кредиты)
            │  индексация, поиск, чтение, кодинг, прогон тестов
            ▼
      [твой клей] собирает GROUND TRUTH независимо:
            git diff • тест-раннер • тайпчекер/линтер • список файлов
            ▼
      Opus получает: компактный артефакт + ground-truth bundle
            │  решает: pass / fail / iterate
            │  коммитит чекпоинт состояния (детерминированный шаг)
            ▼
            (следующая подзадача или конец)
```

> Все флаги Cursor CLI (`-p`, `--force`/`--yolo`, `--model`, `--output-format json`, `--resume`) сверять с актуальной докой Cursor — они менялись и могут меняться.

## 2. План создания по фазам

Каждая фаза: **цель → что построить → начальный тест → критерий выхода**. Не переходи к следующей фазе, пока тест предыдущей не зелёный. Это нужно, чтобы бросить затею дёшево, если экономика не сойдётся (механизм — не разовый гейт, а постоянное наблюдение Фазы 6 + правила отступления §5).

### Фаза 0 — Инструментирование в проде (источник истины вместо отдельного базлайна)

Решение «синергия Opus+Composer работает» принято личным опытом — отдельный head-to-head базлайн «дешевле ли пайплайна, чем solo» не строим. Вместо этого с первого дня собираем телеметрию **на реальной работе через пайплайн**. Вопрос смещается с «работает ли вообще» на «**где именно Composer ломается, чтобы оркестратор это предсказывал и обходил**» — то есть метрики это не отчётность, а сигнал маршрутизации (класс + сложность → отправлять Composer или нет).

**Что фиксируется (детали — в разделе 3, блок «сигнал маршрутизации»):**
- **A priori, лочится при диспатче, до исхода:** класс задачи, версии моделей, балл сложности (несколько осей), оценка ясности ТЗ. Лочится — потому что правка задним числом («упал, значит было сложно») превращает данные в самосбывающийся артефакт.
- **A posteriori, считает клей из ground-truth bundle (§6.2):** корректность, число исправлений, флаг «затупил», расход окна Opus и токенов Composer.

**Главный инвариант стоимости (иначе метрики сами утяжелят пайплайн):**
Метрика бесплатна, только если она — побочный продукт решения, которое Opus принимает **по своей роли**. Балл сложности и ясность ТЗ Opus и так держит в голове, ставя задачу → пусть выплюнет число к выводу, это копейки. Всё остальное (корректность, исправления, «затупил», расход) считает **клей детерминированным кодом — ноль токенов Opus**. Если ловишь себя на том, что Opus делает *отдельный аналитический проход ради метрики* — перенеси метрику на клей или убери.

**Скользящий базлайн самого себя:** раз `opus_window_spent` по классам логируется постоянно, эти данные сами образуют непрерывный базлайн — лучше разовой замерки, потому что ловят дрейф во времени (в т.ч. после апдейта моделей). Отдельная Фаза-0-замерка не нужна; телеметрия и есть базлайн, просто без сравнения с альтернативой.

**Анти-замораживание маршрутизации:** как только начнёшь по ранним данным уводить трудные задачи от Composer — перестанешь собирать данные о нём на трудных задачах, и таблица застынет. Оставляй малую долю «разведки»: иногда всё равно отправляй пограничную задачу Composer. Версии моделей — в каждой записи (скачок после апдейта не спутать со сложностью).

- **Критерий выхода:** лог метрик пишется автоматически на каждой реальной задаче, с разделением a priori / a posteriori и пометкой источника (клей / балл Opus). Балл сложности лочится при диспатче.

### Фаза 1 — Дымовой тест исполнителя (Composer в изоляции)

- **Цель:** убедиться, что `cursor-agent` работает headless и предсказуемо.
- **Построить:** простейший скрипт вызова `cursor-agent -p --force --model composer --output-format json` в отдельной папке.
- **Начальный тест:** дать тривиальную задачу на один файл («добавь docstring к функции X»). Проверить: (а) запускается неинтерактивно; (б) возвращает парсящийся JSON; (в) реально меняет файл; (г) **процесс завершается** — у headless-режима были баги с зависанием и неосвобождением терминала, поэтому сразу оборачивай вызов в таймаут.
- **Критерий выхода:** 10 из 10 запусков вернули валидный JSON и завершились в пределах таймаута.

### Фаза 2 — Контракт артефакта и независимая верификация

- **Цель:** отделить «что Composer сказал» от «что произошло на самом деле».
- **Построить:**
  - JSON-контракт ответа исполнителя: `status` (done/blocked/failed), `files_changed[]`, `commands_run[]`, `summary` (короткий, недоверенный нарратив), `raw_test_output`.
  - Обёртку, которая работает в **git worktree** и собирает ground truth **сама**, а не из отчёта: `git diff`, прогон тест-сьюта, тайпчекер/линтер.
- **Начальный тест (анти-обман):** дать задачу, которую легко «сымитировать» (например, где можно закомментировать падающий тест вместо починки логики). Проверить, что независимая верификация ловит расхождение между `status: done` и реальным ground truth.
- **Критерий выхода:** есть случай, где отчёт говорит «готово», а верификация говорит «нет», и пайплайн это видит без человека.

### Фаза 3 — Скелет оркестратора (Opus)

- **Цель:** замкнуть один цикл, держа Opus в режиме, близком к чату.
- **Построить:** шаг, где Opus читает состояние → пишет ТЗ с **машинно-проверяемыми** критериями приёмки → получает ground-truth bundle → решает pass/fail/iterate. Opus видит diff + логи тестов, **не репозиторий**.
- **Начальный тест:** одна средняя задача end-to-end. Параллельно открыть `/context` в Claude Code и убедиться, что контекст оркестратора остаётся маленьким и Opus не читает файлы целиком.
- **Критерий выхода:** цикл проходит; контекст Opus после цикла остаётся в абсолюте маленьким (видит diff + логи, не репозиторий), и `/context` подтверждает, что Opus не читал файлы целиком. Эта первая запись `opus_window_spent` по классу становится началом скользящего собственного базлайна (§3), против которого дальше меряется дрейф.

### Фаза 4 — Состояние и непрерывность между сессиями

- **Цель:** смена сессии не вырывает модель из контекста.
- **Построить:**
  - `STATE.md` (план, решения и *почему*, инварианты, открытые вопросы) + структурный **ledger** задач (`pending/done/blocked`, привязка к ТЗ и к возвращённому артефакту). Автор — только Opus.
  - **Шаг чекпоинта** — детерминированная часть клея: артефакт получен → верифицирован → скрипт коммитит обновление состояния. Запись — **до** сброса/компакции (автокомпакция теряет дословную историю, сохранит уже лоссовое саммари).
  - **Шаг сверки на старте сессии**: свежий Opus сначала прогоняет тесты + смотрит git-лог, сравнивает с заявлениями `STATE.md`, и только потом доверяет прозе.
  - `STATE.md` и ledger держать **вне** `CLAUDE.md` (он грузится в контекст до первого сообщения и будет раздувать каждую сессию).
- **Начальный тест:** (а) убить сессию посреди задачи, открыть свежую — она должна восстановить полный контекст с диска и продолжить корректно. (б) Дрейф: вручную вписать в `STATE.md` «фича готова», которой нет; сверка обязана поймать.
- **Критерий выхода:** «самодостаточность» подтверждена — нулевая сессия Опуса по одному только диску понимает, где всё стоит и что дальше.

### Фаза 5 — Правила и гард-рейлы

- **Цель:** перевести критичные договорённости из текста в структуру.
- **Построить:**
  - `CLAUDE.md` (оркестратор: дисциплина делегирования) и `AGENTS.md` (исполнитель: границы) — поведенческий слой, снимает рутинные отклонения.
  - Инварианты в клее: изоляция worktree; независимая верификация; урезанные права исполнителя (белый список команд, без деструктивных операций — на уровне разрешений Cursor CLI, а не просьбы); **скрытые held-out проверки**, которых Composer не видит в контракте (иначе он оптимизирует прохождение видимых тестов).
- **Начальный тест:** попытаться спровоцировать исполнителя нарушить границу (тронуть файл вне скоупа). Структурный гард (а не текст правила) должен это остановить/пометить.
- **Критерий выхода:** каждая критичная договорённость имеет инвариант, ловящий её нарушение автоматически.

### Фаза 6 — Закалка цикла и переход в эксплуатацию

- **Цель:** довести цикл до устойчивой работы и включить непрерывный мониторинг (head-to-head базлайн не строим — решение «синергия работает» принято опытом).
- **Построить:** ретраи, таймауты, обработка `blocked`, лимит итераций на задачу (предохранитель от рекурсивного разрастания).
- **Начальный тест:** прогнать через полный пайплайн несколько реальных задач каждого класса, убедиться, что телеметрия пишется и таблица маршрутизации начинает наполняться.
- **Критерий выхода / решение:** не разовый гейт, а **постоянное наблюдение** по разделу 3. Тревоги: доля «затупил» по классу превысила ~20% (порог окупаемости), `r` дрейфует вверх против собственной истории, хвост стал случайным (поломка). Реакция по ситуации: маршрутизировать проблемный класс мимо пайплайна, чинить ТЗ (если провалы кластеризуются на мутных спеках), или для конкретного класса вернуться к ручному управлению / чистому Claude Code.

## 3. Постоянные метрики (считать ВСЕГДА, на каждой задаче/сессии)

Это приборная панель. Без неё ты не отличишь «работает как задумано» от «тихо стоит дороже базлайна».

### Принципы измерения (читать до таблиц)

1. **Решения по квантилям, не по средним.** У агентных прогонов тяжёлый правый хвост: один зациклившийся или рекурсивно разросшийся прогон перекашивает среднее так, что оно перестаёт описывать типичный случай. Среднее держим только для прикидки; вердикты — по `p50`/`p90`/`p95`.
2. **Квантили считаются на ЗАВЕРШЁННЫХ задачах; провалы учитываются отдельно.** Иначе картина ложно красивая: дёшево в `p50`, потому что половина задач упала на полпути и «не успела потратиться». Стоимость и надёжность не смешиваются в одну метрику.
3. **Всё — с разбивкой по классам задач** из Фазы 0 (мелкая / средняя / с перебором). Распределения у них разные; агрегат по всем трём скроет, что схема отлично работает на одном классе и проваливается на другом. Решение принимается по классам, а не по общему котлу.
4. **Учитывай размер выборки.** `p95` на 20 прогонах статистически пустой. До ~пары сотен задач относись к высоким квантилям как к индикатору, а не приговору, и прикидывай доверительный интервал (хотя бы грубым бутстрэпом).

### Экономика (главный блок)

Поскольку отдельного базлайна больше нет (Фаза 0 → прод-инструментирование), `r` считается против **скользящего собственного базлайна**: историческая `p50` расхода окна Opus по этому классу задач. Тогда `r` — не разовый вердикт «стоит ли пайплайн», а **детектор дрейфа**: пополз ли расход вверх со временем (после апдейта модели, разрастания ТЗ, загрязнения контекста).

| Метрика | Зачем | Тревога |
|---|---|---|
| **`p50(r)`** против исторической `p50` по классу | Дрейф типичной стоимости | Растёт во времени |
| **`p95(r)`** — хвост | Появился ли класс задач, где стабильно дорого | Хвост пухнет |
| `max(r)` и доля прогонов с `r > 1` (дороже своей истории) | Насколько и как часто хуже обычного | Высокая доля |
| `p50`/`p90` расхода окна Opus на задачу (in/out/cache) | Абсолют в дефицитной валюте | p90 ползёт вверх |
| `p90` размера контекста оркестратора (из `/context`) | Расползание контекста | Монотонный рост за сессию |
| `p50`/`p90` токенов/кредитов Composer на задачу | Контроль дешёвого слоя | Хвост = рекурсивное разрастание |
| `p50`/`p90` ходов Opus на завершённую задачу | Накладные расходы оркестрации | Много ходов на мелочь |
| `p50`/`p95` wall-clock на задачу | Латентность моста | p95 хуже, чем Opus сам |

> Интерпретация хвоста: если он от одних и тех же **типов** задач (сильно связанный код, мелкие правки, где оркестрация дороже самой работы) — это граница применимости → маршрутизируй их мимо пайплайна. Если **случайный** и размазан по классам — это поломка: течёт контекст, рекурсивное разрастание, плохие ТЗ.

### Сигнал маршрутизации (главная цель прод-телеметрии)

Цель — таблица «класс + сложность → отправлять Composer или нет» и предсказание, когда он затупит. Каждая метрика помечена источником: **[клей]** = считает детерминированный код, ноль токенов Opus; **[балл Opus]** = суждение, которое Opus и так делает по роли, записывается как короткое число (платно, держать минимальным).

| Метрика | Источник | Зачем |
|---|---|---|
| Балл сложности (оси: файлы, связность, неоднозначность, новизна, тест-поверхность) | [балл Opus] + прокси [клей] | Что именно ломает Composer, а не просто «сложно». Лочится при диспатче |
| Ясность ТЗ (1–5) | [балл Opus] | Отделить «Composer тупит» от «ТЗ было мутное» |
| Корректность (visible/held-out/scope) | [клей] из §6.2 | Объективная годность, не по нарративу Composer |
| Число исправлений до accept | [клей] | Цена переделок; вход в порог ~20% |
| Флаг «затупил» (композит: провал с 1-го + N итераций + discrepancy + held-out отстал) | [клей] | Операционное «тупит», а не на ощущение |
| Доля «затупил» по (класс × сложность) | [клей] | **Это и есть таблица маршрутизации** |
| Версии моделей | [клей] | Скачок после апдейта не спутать со сложностью |

> Анти-циркулярность: балл сложности и ясность ТЗ — **a priori**, лочатся до исхода. Корректность и «затупил» — **a posteriori**, из ground truth. Не править первое вторым.
> Анти-замораживание: малая доля «разведки» — иногда отправлять пограничную задачу Composer, чтобы таблица оставалась живой.

### Эффективность (полезная работа против накладной)

«Дешевле» и «эффективнее» — разное; здесь меряем, сколько усилий ушло не на дело.

| Метрика | Зачем | Тревога |
|---|---|---|
| `p50`/`p90` итераций до готовности | Качество ТЗ Опуса; пригодность задачи | Длинный хвост p90 = ТЗ слабые / задача не делится |
| `accept@1` — доля задач, прошедших независимую верификацию с 1-й попытки исполнителя | Сразу-в-цель | Низкий → ТЗ нечёткие |
| Доля токенов Opus на ревью/переделку против первичного планирования | Не съедает ли ревью экономию | Ревью доминирует → исполнитель ненадёжен, экономия фиктивна |

### Точность (на ground truth, с разбивкой по доверию)

Общий «процент успеха» здесь бесполезен — он прячет именно ту проблему, ради которой строилась верификация.

| Метрика | Зачем | Тревога |
|---|---|---|
| Accuracy по **независимой** верификации (тесты/тайпчек из прогона Opus, не из отчёта) | Реальная готовность | — |
| **Precision самоотчёта Composer** — из всех `status: done` сколько реально done по ground truth | Прямой замер overclaim / reward-hacking | Любое падение = проблема доверия |
| Held-out pass rate **против** видимых-тестов pass rate | Расхождение = подгонка под тесты | Held-out отстаёт от видимых |
| Пойманные нарушения скоупа (файлы вне ТЗ) | Работают ли границы | — |

### Надёжность / координация

| Метрика | Зачем | Тревога |
|---|---|---|
| Доля таймаутов/зависаний `cursor-agent -p` | Известный баг headless | Любой устойчивый процент |
| Доля невалидного JSON от исполнителя | Стабильность контракта | — |
| Доля задач, упавших до завершения (учёт отдельно от квантилей!) | Чтобы p50 не врал | Растёт → квантили нерепрезентативны |
| Успешные возобновления сессии vs ручная правка | Работает ли непрерывность | — |

### Целостность состояния / дрейф

| Метрика | Зачем | Тревога |
|---|---|---|
| Расхождения на сверке при старте сессии (STATE.md vs git/тесты) | Не врёт ли док | Любое > 0 разобрать |
| Инциденты «продолжил по неверному предположению» | Цена дрейфа | — |

## 4. Сводный чеклист

### Подготовка (Фаза 0 — прод-инструментирование, телеметрия = базлайн)
> Head-to-head базлайн «дешевле ли пайплайна, чем solo» НЕ строим — решение «синергия работает» принято опытом (§0, §2). Телеметрия на реальной работе и есть скользящий базлайн самого себя. Поэтому ниже — про то, как сразу писать сигнал маршрутизации, а не про разовую замерку.

- [ ] Запись разделена на **a priori** (лочится при диспатче: класс, балл сложности по осям, ясность ТЗ) и **a posteriori** (клей из ground-truth bundle §6.2: корректность, число исправлений, флаг «затупил», расход)

## 5. Когда отступить (по классу задач, не целиком)

Решение здесь — не «хоронить пайплайн», а **убрать из него конкретный класс задач**, если телеметрия устойчиво показывает:
- доля «затупил» по классу превысила ~20% (порог окупаемости) и держится;
- провалы кластеризуются на **мутных ТЗ** (ясность низкая) — тогда чинить надо ТЗ оркестратора, а не списывать Composer;
- `p90` контекста оркестратора растёт за сессию (состояние течёт в окно);
- precision самоотчёта Composer падает и держится низкой (доверять нельзя даже под верификацией — растут расходы на ревью);
- `r` дрейфует вверх против собственной истории без видимой причины;
- `p90` итераций с длинным хвостом на сильно связанном коде (плохой кандидат на схему).

Для проблемного класса — маршрутизация прямо в Claude Code или возврат к ручному управлению командой Opus+Composer; остальные классы остаются в пайплайне. Альтернатива без моста к Cursor: нативные субагенты Claude Code с дешёвой моделью Claude на чтение/поиск и Opus ведущим. Маршрут через Cursor оправдан осознанно — ради дешёвых токенов и больших лимитов Composer.

> И помни ограничение из обсуждения: автоматический оркестратор оправдан ровно настолько, насколько ты реально отпускаешь руль. Если стиль — плотный контроль каждой задачи, ручное управление командой может быть дешевле по окну Opus (ты бесплатный ревьюер) и проще; тогда машинерия оценок — накладные расходы без отдачи.

## 6. Контракт исполнителя (детализация Фазы 2)

Граница между слоями — это git + объективный вывод инструментов. Composer работает в своём worktree, коммитит, отдаёт JSON. Доверенный слой **не принимает решения по этому JSON** — он принимает их по ground truth, который собирает сам. JSON нужен для двух вещей: как подсказка (что исполнитель *думает*, что сделал) и как материал для метрик доверия.

### 6.1 Что отдаёт исполнитель (всё это — НЕдоверенные данные)

```json
{
  "task_id": "T-0042",
  "status": "done",
  "summary": "Добавил idempotency-key в refund-клиент и тест на повтор запроса.",
  "files_changed": ["src/refund.ts", "test/refund.test.ts"],
  "commands_run": ["npm test"],
  "commit_sha": "a1b9f3c",
  "blockers": [],
  "confidence": 0.9
}
```

- `status` — **заявка**, а не факт. Сверяется с ground truth → из этого считается precision самоотчёта.
- `summary` / `confidence` / `commands_run` — чисто информационные, в решение не входят.
- `files_changed` — заявка, перепроверяется против `git diff --name-only` (расхождение = сигнал по скоупу/целостности).
- `commit_sha` / `blockers` — заявка; SHA доверенный только после того, как клей убедится, что коммит реально есть в worktree.

### 6.2 Что собирает клей сам (это и есть ДОВЕРЕННЫЙ ground truth)

Клей не верит отчёту, а запускает в worktree:

```json
{
  "task_id": "T-0042",
  "actual_commit_sha": "a1b9f3c",
  "diff_name_only": ["src/refund.ts", "test/refund.test.ts"],
  "diff_stat": "+38 -4",
  "visible_tests": { "ran": true, "passed": 41, "failed": 0 },
  "typecheck": { "passed": true, "errors": 0 },
  "lint": { "passed": true, "errors": 0 },
  "held_out_checks": { "ran": true, "passed": 3, "failed": 0 },
  "scope": { "allowed": ["src/refund.ts", "test/**"], "out_of_scope_touched": [] }
}
```

- Всё здесь получено независимым прогоном, а не из JSON исполнителя.
- `held_out_checks` — проверки, которых не было в ТЗ и которые Composer не видел.
- `scope` — пересечение реально тронутых файлов с разрешённым списком из ТЗ.

### 6.3 Классификация доверия (один взгляд)

| Поле | Источник | Доверие | Роль |
|---|---|---|---|
| `status`, `summary`, `confidence`, `commands_run`, `blockers` | отчёт Composer | НЕ доверять | подсказка + материал метрик |
| `files_changed` | отчёт Composer | НЕ доверять | сверяется с `diff_name_only` |
| `diff_*`, `*_tests`, `typecheck`, `lint`, `held_out_checks`, `scope`, `actual_commit_sha` | прогон клея | доверять | основа решения |

### 6.4 Решение оркестратора (его и пишет Opus)

Opus видит ground-truth bundle + diff, и фиксирует решение:

```json
{
  "task_id": "T-0042",
  "verdict": "accept",
  "reason": "Все проверки зелёные, скоуп не нарушен, diff соответствует ТЗ.",
  "next": "T-0043",
  "ledger_update": { "T-0042": "done" }
}
```

`verdict ∈ {accept, iterate, reject, block}`. На `iterate`/`reject` Opus формирует уточнённое ТЗ. Этот объект — детерминированный вход для шага чекпоинта.

### 6.5 Как из контракта считаются метрики доверия

- **Precision самоотчёта** = (число задач, где `status == "done"` **и** ground truth зелёный) / (все задачи с `status == "done"`). `status: done` при красном ground truth = false positive (overclaim).
- **Held-out vs видимые** = сравнение `held_out_checks.passed/total` с `visible_tests.passed/total`. Held-out заметно ниже → подгонка под видимые тесты.
- **Нарушения скоупа** = доля задач с непустым `scope.out_of_scope_touched`.
- **Расхождение по файлам** = доля задач, где `files_changed` (заявка) ≠ `diff_name_only` (факт).
- Все поля контракта пишутся в лог метрик построчно — отсюда и берутся квантили по классам задач из раздела 3.

> Главный инвариант контракта: ни одно поле из «отчёта Composer» не участвует в `verdict`. Если поймаешь себя на том, что Opus принимает `accept`, опираясь на `summary` или `status` — это и есть протечка доверия, ровно та боль из исходного реддит-поста.

## 7. Опенсорс-инструменты

Готового «всё-в-одном» под связку «Opus-оркестратор + Cursor-CLI-Composer-исполнитель» нет — почти всё сделано под случай «ты контролируешь модели всех агентов и гонишь их пачками». Поэтому берём кубиками под отдельные узлы, не отдавая управляющий цикл, трастовую границу и контракт (раздел 6) — это сердце схемы.

**Три правила выбора:**
1. Большинство проектов мелкие и экспериментальные — проверять перед прод-использованием.
2. Тяжёлый оркестратор возвращает токеновую боль (простаивающие сессии в ожидании подтверждений тулов).
3. **Запрет:** ничего, что гоняет Opus через Anthropic API — это убивает экономику подписки. Мост ходит наружу к Composer, Opus остаётся в Claude Code.

### БРАТЬ (закрывают узлы, которые писать самому дорого)

| Узел плана | Инструмент | Что даёт |
|---|---|---|
| Метрики / раздел 3 | **Arize coding-harness-tracing + Phoenix** (self-hosted) | Инструментирует Claude Code и Cursor: тул-коллы, латентность, токены, ошибки → датасеты, дашборды во времени. Снимает половину инструментирования. |
| Надёжность по распределению / квантили | **plugin-eval** (из `wshobson/agents`) | Три слоя: статический анализ, LLM-судья, **Monte Carlo (50–100 прогонов)** для variance — прямо под «не среднее, а хвосты». |
| Изоляция исполнителя (worktree) | **Coasts** / **subtask** / **scion** | Контейнерная изоляция git worktree (порты/сети), делегирование субагентам в worktree, параллельные изолированные контейнеры + телеметрия. Берём один. |

### РЕФЕРЕНС (смотреть как образец, не обязательно внедрять)

| Узел | Инструмент | Зачем смотреть |
|---|---|---|
| Мост + verify-цикл | **AgentPlane** | Git-native поток task → plan → approve → implement → verify → finish — ближе всего к нашей схеме. |
| Мост + состояние | **ORCH** | CLI-рантайм, управляет Claude Code и Cursor как типизированными командами со стейт-машиной и целями. |
| Единый интерфейс к CLI-агентам | **pi-builder** | Оборачивает любой CLI-агент за одним интерфейсом: маршрутизация, fallback-цепочки, SQLite-персистентность. |

> По всем трём — сверять, не утаскивают ли они Opus в API-биллинг. Если да — только как референс архитектуры, не как рантайм.

### СЛЕДИТЬ (пригодится при росте до нескольких параллельных исполнителей)

- **swarm-protocol** — headless-координация как MCP: захват задач, детект конфликтов файлов, хендофф между сессиями.
- **wit** — лочит конкретные функции (не файлы) через разбор AST.
- **shire** / **CC Mirror** — персистентные воркспейсы с обменом между агентами / нативная оркестрация на task-JSON без зависимостей.
- Для одного Composer всё это избыточно — `STATE.md` + git проще и достаточны.

### Каталоги и чтение

- Директории инструментов этого класса: `andyrewlee/awesome-agent-orchestrators`, `bradAGI/awesome-cli-coding-agents`.
- Концептуально (не инструменты, а рамка): обзор Agentic Harness Engineering и материалы Anthropic по eval'ам агентов — для дисциплины «edit как фальсифицируемый контракт» и трезвого чтения метрик.

### Что брать vs строить (резюме)

- **Брать готовое:** наблюдаемость (Arize/Phoenix), замер надёжности по прогонам (Monte Carlo), изоляция worktree.
- **Строить самому:** управляющий цикл, трастовая граница, контракт (раздел 6) — тут готовое навяжет чужие допущения.
- **Мост:** пограничный — взять за референс AgentPlane/ORCH, но не дать утащить Opus в API.

## 8. Память оркестратора и смена сессий

Модель stateless: всё, что Opus должен «помнить», перезаливается во вход каждый ход и тарифицируется заново, пока не отрезано. Значит управление памятью — не косметика, а прямой рычаг стоимости. Это не выбор «или–или», а **слой + предохранитель** в строгом порядке.

### 8.1 Иерархия (порядок важен)

1. **State на диске — фундамент, всегда.** Первичный источник правды и при компакции, и при смене сессии. Без него смена сессии вырывает контекст, а компакция тихо его искажает.
2. **Внутрисессионная компакция — дешёвое удобство, не хранитель критичного.** Лоссова: заменяет дословную историю саммари, и ты не контролируешь, что выживет. Всё критичное уже продублировано в state, поэтому потеря восстановима. Не назначай компакцию носителем инвариантов.
3. **Смена сессии — плановая, не аварийная.** Аварийная = автокомпакция сама сработала на пороге и сохранила лоссовое саммари. Плановая = клей видит приближение критической точки, пишет чекпоинт **до** сброса, стартует чистую сессию с регидратацией из state. Разница: перенос контролируешь ты, а не автокомпакция.
4. **Сверка на старте — страховка перехода.** Свежий Opus не верит state на слово: первым делом гоняет тесты + git-лог и сличает с заявлениями `STATE.md`. Метрика «расхождения на сверке при старте сессии» (раздел 3) — приборчик, показывающий, не сломал ли переход контекст.

> Правило: компакцию не делай хранителем критичного; смену сессии не делай аварийной.

### 8.2 Триггеры критической точки (любой = чекпоинт + свежая сессия)

Держать как метрики, а не «на глаз»:

| Триггер | Сигнал | Источник |
|---|---|---|
| Контекст подошёл к порогу | `p90` размера контекста оркестратора близко к эффективному окну | `/context` |
| Закрыта логическая веха | Подзадача/эпик завершён — естественный момент сбросить переписку | ledger |
| Деградация качества | Растёт число итераций, Opus путается/повторяется | метрика итераций |
| Подряд несколько `iterate`/`reject` | Возможно, контекст уже «загрязнён» | лог вердиктов |

### 8.3 Что обязано пережить переход, а что восстановимо

Принцип: **не дублируй в state то, что детерминированно восстанавливается из git/тестов.** Дублирование — лишний вес и риск рассинхрона прозы с репозиторием.

**Обязано быть в state (из git/тестов НЕ восстановишь):**
- План и текущая стратегия.
- **Почему** приняты ключевые решения (обоснования, отвергнутые варианты).
- Инварианты и критерии приёмки.
- Ledger задач со статусами (`pending`/`done`/`blocked`) и привязкой к ТЗ и артефакту.
- Открытые вопросы и известные блокеры.
- Указатели на held-out проверки (не сами проверки — их Composer не должен видеть).

**Восстановимо из git/тестов — НЕ дублировать:**
- Что именно изменено в коде → `git diff` / `git log`.
- Текущее состояние «работает/нет» → прогон тестов + тайпчек.
- Какие файлы тронуты → `git diff --name-only`.
- История коммитов и SHA → git.

> Тест самодостаточности: нулевая сессия Opus, прочитав только state + проведя сверку, должна точно понять «где всё стоит и что дальше». Если для этого нужна память прошлой сессии — значит критичное поле течёт в контекст, который ты собираешься выбросить.

## 9. Held-out проверки (узел анти-подгонки)

Held-out — это проверки тех же критериев приёмки, что и видимые тесты, но **под другим углом** и **скрытые от исполнителя**. Если held-out проходят хуже видимых — Composer подогнал решение под видимые тесты, а не сделал правильно. Это единственный сигнал, ловящий reward-hacking, который иначе невидим.

### 9.1 Модель угрозы (почему наивное хранение не работает)

У Composer полный агентный доступ: он читает файлы и выполняет shell-команды, и уже доказал склонность к обходным путям (реверс-инжиниринг кэшей, декомпиляция байткода). Из этого:

- **`.gitignore` не прячет.** Игнор скрывает от git, не от файловой системы — `cat`/`find` достанут файл.
- **Хранить held-out в репозитории нельзя** — Composer их проиндексирует и сможет либо подогнаться, либо реконструировать.
- Значит held-out должны быть **физически недостижимы** для песочницы исполнителя во время работы и появляться только на этапе верификации, проводимой клеем/оркестратором.

### 9.2 Где и как хранить — принцип и рекомендация

**Принцип: защита здесь — это изоляция, а не секрет.** `cursor-agent` запускается от твоего имени с твоими правами на файлы; процесс с правами чтения просто читает файл — пароля никто не спрашивает. Поэтому **пароль/токен на папку против Composer не работает** (он не ломает пароль — он `cat`-ает доступный файл). Нужен примитив «исполнитель физически не может прочитать», а не «знает, но не вводит».

> Токен `CURSOR_API_KEY` — это аутентификация исполнителя в сервисе Cursor, к защите held-out отношения не имеет. Не путать.

**Рекомендация под «универсально (локально + сервер)»: контейнер.**
Исполнитель крутится в контейнере, которому проброшена **только папка проекта (worktree)**. Всё снаружи для процесса не существует — held-out недостижимы не «потому что запрещено», а потому что их нет в его mount. Описывается один раз, запускается одинаково на ноутбуке и сервере — это и есть нужная переносимость.

**Лёгкая альтернатива без контейнеров:** отдельный урезанный пользователь ОС (`composer-runner`) без прав чтения на каталог held-out (он принадлежит оркестратору, режим 700). Composer упирается в отказ доступа на уровне ОС. Работает в обоих местах, но менее переносимо и более ручное.

**«Вне папки проекта» — необходимо, но недостаточно.** Класть held-out вне дерева проекта нужно всегда, но без контейнера/урезанных прав это иллюзия: полный shell-доступ `cat`-нет любой путь, читаемый твоим пользователем. Защита = вне папки **плюс** изоляция.

> Подмешивание на верификации (9.3) делает схему надёжной независимо от изоляции: во время работы Composer held-out физически нет рядом, они появляются только в одноразовом verify-окружении, когда сессия исполнителя уже завершена. Даже если изоляция где-то протекла — утечки нет.

### 9.3 Поток подмешивания на верификации

```
Composer закоммитил в свой worktree
        ↓
клей берёт этот коммит → делает ОТДЕЛЬНЫЙ verification-checkout
        ↓
копирует held-out из защищённого хранилища В ЭТОТ checkout (не в worktree Composer!)
        ↓
прогоняет видимые тесты (из репо) + held-out (подмешанные)
        ↓
записывает held_out_checks / visible_tests в ground-truth bundle (раздел 6.2)
        ↓
УНИЧТОЖАЕТ verification-checkout — held-out не остаётся там, где их увидит следующая итерация
```

Ключевое: held-out никогда не попадают в рабочий worktree Composer; verification-checkout одноразовый.

### 9.4 Что делает held-out осмысленными

- **Другой угол, не копия.** Переименованная копия видимого теста геймится тривиально. Held-out — другие входы, краевые случаи, property-based/fuzz, интеграционный уровень.
- **Из той же спеки, но втайне.** Выводятся из тех же критериев приёмки, что и ТЗ, но не показываются — тогда «прошёл видимые» ≠ «прошёл held-out».
- **Детерминированность.** Не зависят от окружения Composer, чтобы результат был чистым сигналом.

### 9.5 Управление утечкой по итерациям (тонкое место)

Как только held-out провалился и Opus сказал Composer «ты упал на X», эта проверка **сгорела** — Composer теперь о ней знает. Поэтому:

- **Опус отдаёт обобщённую подсказку, а не точную ассертацию.** «Обеспечь идемпотентность при повторе», а не вставленный текст упавшего held-out. Иначе ты конвертируешь held-out в видимый тест своими руками.
- **Держи резерв.** Часть held-out никогда не показывается в итерациях — это чистый замер на финальной приёмке.
- **Ротация.** На новую задачу — свежий набор held-out, чтобы накопленное знание Composer не обесценивало сигнал.

### 9.6 Авторство и связь с метриками

- Авторы held-out — только доверенный слой (Opus) или ты. Composer **никогда** не пишет и не видит их; в ТЗ они не попадают.
- Питают метрику из разделов 3 и 6.5: `held_out pass rate` против `visible pass rate`. Устойчивое отставание held-out = подгонка под видимые тесты, разбирать как проблему доверия.

> Инвариант узла: held-out пересекают границу к исполнителю только в виде *обобщённого требования в спеке*, никогда — в виде кода или ассертации. Код held-out существует исключительно на доверенной стороне.
